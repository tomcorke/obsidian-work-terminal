# Release Process

[Back to README](../README.md)

## Goals

Every GitHub release and `CHANGELOG.md` entry should use the same reader-friendly structure. Do not publish the raw autogenerated PR list as the final release notes.

Use these sections, in this order, whenever they are relevant:

1. Highlights
2. Improvements
3. Fixes
4. Upgrade notes
5. Full changelog

## Style rules

- Group changes by user outcome, not by PR number.
- Lead with the 3-4 changes users will notice most.
- Keep bullets short, concrete, and scan-friendly.
- Put removals, migrations, and operational caveats in **Upgrade notes**.
- End every release with a compare link or history link.
- Omit empty sections instead of padding them with filler.
- Rewrite old releases if needed so the published GitHub release body and `CHANGELOG.md` stay aligned.

## Release note template

```md
## Highlights
- ...

## Improvements
- ...

## Fixes
- ...

## Upgrade notes
- ...

## Full changelog
- Compare with <previous-tag>: https://github.com/tomcorke/obsidian-work-terminal/compare/<previous-tag>...<new-tag>
```

## Workflow

### 1. Draft the notes in `CHANGELOG.md`

- Add the new version section using the headings above.
- Rewrite commit subjects into user-facing summaries.
- Update older entries too if their formatting has drifted.

### 2. Bump version metadata

Update all release-version files together:

- `manifest.json`
- `package.json`
- `versions.json`

### 3. Validate from a worktree

The project root is the live plugin checkout. Do not switch branches there.

Run validation from a clean worktree:

```bash
pnpm exec vitest run
pnpm run build
```

Commit the changelog, docs, and version bump on a branch, then merge that branch before tagging.

### 4. Tag without touching the project root

From any clean worktree after the release branch has been merged:

```bash
git fetch origin
VERSION=0.3.0
git tag "$VERSION" origin/main
git push origin "$VERSION"
```

Tagging `origin/main` directly avoids switching the root checkout away from the user's live plugin files.

### 5. Publish or update the GitHub release

After the tag is pushed:

1. Wait for the release workflow to attach `main.js`, `manifest.json`, `styles.css`, and `pty-wrapper.py`.
2. Copy the matching `CHANGELOG.md` entry into a temporary file.
3. Publish the formatted notes with `gh`.

If the release already exists:

```bash
gh release edit "$VERSION" \
  --repo tomcorke/obsidian-work-terminal \
  --title "$VERSION" \
  --notes-file /tmp/"$VERSION"-notes.md
```

If the release does not exist yet:

```bash
gh release create "$VERSION" \
  --repo tomcorke/obsidian-work-terminal \
  --verify-tag \
  --title "$VERSION" \
  --notes-file /tmp/"$VERSION"-notes.md
```

## Release checklist

- [ ] `CHANGELOG.md` entry written in the standard format
- [ ] `manifest.json`, `package.json`, and `versions.json` updated
- [ ] `pnpm exec vitest run` passed
- [ ] `pnpm run build` passed
- [ ] Release branch merged
- [ ] Tag pushed from a worktree
- [ ] GitHub release body matches the changelog entry
- [ ] Release assets are attached
